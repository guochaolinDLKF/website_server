## 1. 项目背景分析

### 1.1 项目定性（基于真实代码）

本项目是一套 **命理推算 / 八字算命系统的后端服务**（Java 包名 `com.fortunetelling`，数据库名 `zhouyi`），为移动端 / PC 端的算命类应用提供接口支撑。**当前仅有面向 C 端用户的业务接口，尚无任何运营后台（管理端）实现**。

> ⚠️ 注意：[项目图片分析策划案.md](项目图片分析策划案.md) 描述的是一套「游戏数据分析 BI 看板」原型，其业务域（游戏：玩法/广告变现/金币钻石经济）与本项目（算命：八字/会员权益购买）**并不一致**。本策划案以**当前真实项目**为准进行设计，并在第 12 章说明二者如何取舍衔接。

### 1.2 技术栈（真实）

| 类别 | 技术 |
|------|------|
| 后端框架 | Spring Boot 3.1.5 / JDK 21 |
| 数据库 | MySQL 8.0（库名 `zhouyi`） |
| ORM | MyBatis-Plus |
| 缓存 | JetCache + Redis（0 号库）+ Caffeine |
| 认证 | Sa-Token（C 端用户态，含自定义 Token 解析 `CustomTokenParser`/`CustomTokenInterceptor`） |
| 连接池 | Druid |
| 消息队列 | RocketMQ（`rocketmq.enabled` 可开关，当前默认关闭） |
| 定时任务 | XXL-Job（`xxl.job.enabled` 可开关，已有 5 个 JobHandler） |
| 支付 | 微信 V3 / 支付宝 V3 / 银联 / 苹果 IAP（多策略 `PaymentStrategyManager`） |
| 对象映射 | MapStruct |
| API 文档 | Swagger3 / OpenAPI（含 `SwaggerAuthController` 登录保护） |
| 部署 | Docker + SSH/SCP 远程部署 |

### 1.3 关键现状结论（影响后台设计）

1. **C 端用户体系完整**：用户、八字记录、断事笔记、标签、设置、运势记录、订单、支付、退款、用户权益均已落库。
2. **后台权限骨架已存在但未落地**：`Role` / `Permission` / `RolePermission` / `UserRole` 四个实体类**已存在于代码中**，但**数据库无对应建表脚本**，也**无任何管理端控制器 / 服务**。→ 运营后台可在此骨架基础上补全。
3. **缺少后台管理员体系**：无 `admin_user` 表、无后台登录接口（`AuthController` 标注「暂未使用」，且面向 C 端用户）。
4. **埋点仅有元数据、无事实数据**：`custom_event` / `custom_event_property` 仅存储「事件定义」，`/game/custom_data_analytics` 仅写入事件定义，**没有存储用户实际行为事件的事实表**，也**没有用户登录日志表**。→ 活跃 / 留存 / 功能使用类分析缺少数据底座（见第 10 章新增项）。

以下 4 个实体**已在代码中定义**，但数据库无对应表、无管理端逻辑。运营后台应**复用并补全**它们，而非另起炉灶：

| `UserRole` | `user_role` | 用户-角色关联（后台将改为「管理员-角色」语义） |

- **类型 3（高级设置/内容）**：神煞设置、五行旺衰设置、干支关系设置、纳音介绍、涂郭讲堂、遁甲自定义、人员司令设置、命宫身宫设置

## 4. 运营后台整体规划

### 4.1 系统定位

在**现有 Spring Boot 工程内新增独立的管理端模块**（建议包路径 `com.fortunetelling.admin.*`，统一前缀 `/admin/**`），与 C 端 `/game`、`/order`、`/pay` 等接口**物理隔离、互不影响**。前端为**独立的 Web 后台单页应用**（建议 Vue3 + Element-Plus + ECharts）。

### 4.2 整体结构

```
运营后台（Web 管理端）
        │
   后台登录系统（admin_user + Sa-Token 独立账号体系）
        ↓
   数据驾驶舱（Dashboard 总览）
        ↓
   用户管理（用户列表 / 详情 / 禁用）
        ↓
   会员与订单管理（订单 / 退款 / 权益 / 商品）
        ↓
   功能配置（items 20 项功能开关/价格/排序）
        ↓
   内容运营（Banner / 文案 / 公告）
        ↓
   运营活动（签到 / 优惠 / 会员活动）
        ↓
   数据分析（用户/收入/功能 多维分析）
        ↓
   系统设置（全局配置 / 接口配置 / 字典）
        ↓
   权限与日志（管理员 / 角色 / 权限 / 操作日志）
```

### 4.3 鉴权方案

- 后台使用**独立的 Sa-Token 登录域**（与 C 端用户态隔离，避免互相挤下线）。建议 `StpUtil` 使用独立 `loginType=admin` 或独立 `StpInterface` 实现。
- 后台所有接口走 `/admin/**`，由独立拦截器校验后台 Token + 基于 `role`/`permission` 的注解鉴权（`@SaCheckPermission`）。

## 5. 功能模块设计

### 5.1 后台登录与权限管理

**管理员登录**
- 功能：账号 + 密码登录、密码加密校验（BCrypt）、Token 下发、登录日志、（可选）图形/短信验证码
- 接口：`POST /admin/auth/login`、`POST /admin/auth/logout`、`GET /admin/auth/info`、`POST /admin/auth/change-password`

**权限系统（复用 Role/Permission 骨架）**
- 角色管理：增删改查、启用/禁用、分配权限
- 权限管理：菜单/按钮/接口三类权限树（`permission_type` 1/2/3）
- 管理员-角色绑定（`user_role` → 管理员维度）

预设角色（详见第 8 章）：超级管理员、运营、客服、财务、数据分析。

### 5.2 数据驾驶舱（首页 Dashboard）

> 指标口径**严格基于真实可取数的表**。标注「★需新增数据」的指标依赖第 10 章新增表，可作为二期。

```
首页 Dashboard：
顶部（核心数据卡片）：
  用户：总用户数(user) | 今日新增(user.createTime) | VIP用户数(user.isVip=1) | ★活跃用户(需登录日志)
  收入：累计收入(payments.SUCCESS) | 今日收入 | 今日订单数(orders) | 有效权益数(user_benefits.is_valid=1)
  转化：付费用户数 | 付费渗透率 = 付费用户/总用户 | VIP占比

中间（趋势图）：
  近30天新增用户趋势(user.createTime)
  近30天收入趋势(payments.payment_time)
  近30天订单量趋势(orders.create_time)

底部（业务排行/分布）：
  热门商品排行(orders 按 item_id 聚合)
  支付渠道分布(orders.pay_source: WECHAT/ALIPAY/APPLE)
  权益类型分布(user_benefits.item_type 1/2/3)
  ★功能使用排行(需事件事实表)
```

### 5.3 用户管理模块

**用户列表**（数据源 `user` 表）
- 展示字段：用户ID、昵称(nickName)、手机号(phoneCode，脱敏)、性别(gender)、是否VIP(isVip)、状态(status)、注册时间(createTime)、更新时间
- 功能：按手机号/昵称/ID 搜索；按 VIP 状态、启用状态、注册时间区间筛选；分页；查看详情；**启用/禁用（改 `user.status`）**；导出
- ⚠️ **禁用而非物理删除**（用户与订单/权益强关联，删除会破坏数据完整性）

**用户详情页**
- 基础信息：账号(phoneCode)、昵称、性别、出生信息(birthProvince/City/Area + birthTime)、头像、注册/更新时间、状态、VIP
- 业务信息（按 user_id 关联）：
  - 八字记录（`eight_record`）
  - 断事笔记（`eight_char_note`）
  - 运势记录（`fortune_record`）
  - 标签分组（`user_tag`）
  - 个性化设置（`setting`）
  - 订单记录（`orders`）
  - 权益记录（`user_benefits`）
  - 短信记录（`sms_send_record`）

### 5.4 会员 / 订单管理模块

> 本项目「会员」即通过购买 `items`/`products` 获得的 `user_benefits` 权益（含 `vip_month` 时长、`expire_time` 到期）。

**商品套餐管理**（数据源 `items` + `products`）
- 字段：商品ID(item_id)、描述(item_desc)、权益类型(item_type)、原价(item_price)、优惠价(item_discount)
- 功能：列表、编辑价格/描述、（如需上下架则见第 10 章新增 `status` 字段）

**订单管理**（数据源 `orders` + `payments` + `refunds`）
- 列表字段：订单号、用户、商品名、金额(原价/优惠价)、订单状态(order_status)、支付状态(payment_status)、支付渠道(pay_source)、创建时间、支付时间
- 功能：多条件查询（订单号/用户/状态/时间/渠道）、订单详情（含支付与回调信息）、导出、状态统计
- 退款管理：退款单列表、退款详情（`refund_items` JSON）、退款状态跟踪
- 异常监控：`payment_callback_failures` 回调失败列表与重试状态

**权益管理**（数据源 `user_benefits`）
- 列表：用户、权益名、类型、购买时间、到期时间、是否有效
- 功能：查询、到期统计、（谨慎）手动调整有效性

### 5.5 功能配置管理

> 对应需求文档「客户端功能后台配置」，数据底座为 `items` 表（已预置 20 项）。

后台可管理：功能开关、功能排序、功能价格、功能描述、功能状态、（可选）使用次数限制。

- **可直接复用**：`items.item_desc`（描述）、`item_price`/`item_discount`（价格）、`item_type`（分组）
- **需扩展字段**（见第 10 章）：`status`（上下架）、`sort_order`（排序）、`func_switch`（开关）、`usage_limit`（次数限制）

### 5.6 内容运营管理（全部为新增）

**Banner 管理**（新增 `banner` 表）
- 功能：上传图片、设置跳转链接/类型、排序、启停、生效时间
**文案 / 公告管理**（新增 `notice` 表）
- 字段：标题、内容、类型、发布时间、状态、排序

### 5.7 资源管理系统（新增）

- 管理头像、Banner 图、配置文件等静态资源（新增 `resource_file` 表）
- 功能：上传、删除、替换、版本/引用记录

### 5.8 运营活动模块（新增）

### 5.9 数据分析模块

> 基于真实数据可落地的分析；缺数据的部分标注并入二期。

- **用户分析**：新增趋势(user.createTime)、VIP 转化、性别/地域分布(birthProvince)、★活跃/留存（依赖新增登录日志）

- **功能分析**：★功能使用次数/排行/转化（依赖新增事件事实表）

### 5.10 系统设置模块（新增）

- **全局配置**（新增 `sys_config` 表）：APP 版本、公告、客服信息、协议地址、开关项
- **接口配置**：第三方 API / 支付参数 / 短信参数的**只读展示或受控管理**（敏感项仍以 yml/密钥管理为准，后台不明文存储密钥）
- **数据字典**：订单状态、支付状态、权益类型等枚举集中维护（可只读映射现有 enum）

## 8. 权限体系设计

### 8.1 角色与权限矩阵

| 角色(role_code) | 用户 | 订单/退款 | 商品/功能配置 | 内容/活动 | 数据分析 | 系统/权限 |
|-----------------|:----:|:--------:|:-------------:|:--------:|:--------:|:--------:|

| OPERATOR 运营 | 查看 | 查看 | 编辑 | 编辑 | 查看 | — |

- 后端接口用 `@SaCheckPermission("xxx")` 校验；前端按菜单/按钮权限渲染

```
      后台登录与权限必须独立账号，不可复用 C 端 user 表。

原因：内容运营（轮播图/公告/文案）当前完全缺失，无任何承载表。

需要新增数据结构：activity
原因：运营活动（签到/优惠/会员活动）当前无承载表。

需要新增数据结构：sys_config
原因：全局配置（版本/公告/客服/协议）当前散落在 yml，运营无法动态调整。

需要新增数据结构：resource_file
原因：资源管理（图片/音视频/上传文件）当前仅有 user.avatar 字段，无统一资源表。

需要新增数据结构：user_login_log
原因：活跃/留存分析需用户登录明细，当前仅有 sms_send_record，无登录日志。

需要新增数据结构：event_log（行为事件事实表）
原因：custom_event 仅为事件「定义」元数据，缺少用户实际行为「事实」数据，
      功能使用次数/排行/转化等分析无数据来源。
      ★ 该表数据量大，建议二期评估是否引入分析型存储（如 ClickHouse），一期可先用 MySQL 分表。

需要新增字段：items 表追加 status、sort_order、func_switch、usage_limit
原因：功能配置需上下架/排序/开关/次数限制，原表无这些运营字段。
      （以 ALTER 追加，不影响客户端现有读取逻辑。）
```

### 10.4 建表注意事项

- 所有新增表统一含 `create_time`、`update_time`、`deleted`（逻辑删除，与现有 MyBatis-Plus `@TableLogic` 一致）。
- 字符集 `utf8mb4`，与现有库一致。
- 新增表不与客户端写入路径耦合，避免影响 C 端稳定性。

## 12. 与《项目图片分析策划案》的衔接说明

[项目图片分析策划案.md](项目图片分析策划案.md) 描述的是**游戏数据分析 BI 看板**，其指标体系可作为本项目「数据分析模块」的**设计参考**，但需按算命 App 实际取舍：

| 图片策划案看板 | 本项目是否适用 | 说明 |
|----------------|:--------------:|------|
| 新增数据 | ✅ 适用 | 基于 `user.createTime`，渠道维度需新增埋点 |
| 活跃数据(DAU/WAU/MAU) | ⚠️ 需新增 | 依赖 `user_login_log`（P2） |
| 留存数据 | ⚠️ 需新增 | 依赖登录日志（P2） |
| 内购数据 | ✅ 高度适用 | 直接复用 `orders/payments/refunds/user_benefits`，可做 ARPU/付费渗透/复购/金额分布 |
| 广告变现数据 | ❌ 不适用 | 算命 App 无游戏广告变现体系，除非新增广告 SDK 数据 |
| 资源数据(金币/钻石) | ❌ 不适用 | 本项目无虚拟货币经济系统 |
| 玩法数据 | ❌ 不适用 | 本项目无关卡/对局玩法 |
| 用户画像 | ⚠️ 部分适用 | 付费分层(大中小R)、生命周期、RFM 可基于订单/权益构建；等级/关卡进度不适用 |

> 结论：**优先落地「内购/收入分析」与「新增分析」**（数据现成）；活跃/留存/画像类作为 P2，需先补登录与行为埋点；广告变现/资源/玩法类**不纳入本项目**（避免臆造不存在的业务）。

## 13. 重要约束与风险提示

1. **不脱离当前项目**：所有模块均基于 `zhouyi` 库真实表设计，新增能力以独立表/独立 `/admin` 接口承载。
2. **优先复用已有数据结构**：用户、订单、支付、权益、商品、八字记录等**直接读取现有表**，不重复建模。
3. **不修改客户端已有逻辑**：后台为只读 + 受控管理；对 `items` 仅 ALTER 追加可空字段，不改变 C 端现有读取；不改动 `/game`、`/order`、`/pay` 等 C 端接口。
4. **不臆造业务**：广告变现、虚拟货币、玩法关卡等本项目不存在的业务**不予设计**；活动发奖等需求待与客户端确认后再细化。
5. **缺失数据已显式标注**：所有「需要新增数据结构 + 原因」集中于第 10 章，便于后续开发评审。
6. **安全**：后台账号独立、密码加密、操作审计；支付密钥/第三方密钥仍由配置与密钥管理，后台不明文存储。
7. **待确认项**：
   - 后台前端技术栈是否定为 Vue3 + Element-Plus？
   - `event_log` 是否引入 ClickHouse，还是一期 MySQL 即可？
   - 运营活动的具体玩法与发奖是否需要客户端联动？
   - 渠道(channel)来源数据如何采集（影响新增/画像分析的渠道维度）？
