## 0. 关键结论速览（务必先读）

1. **后端落点是当前 `server` 项目（包 `com.ydzz`），不是策划案假设的 `com.fortunetelling`。** 策划案是产品/数据设计参考，实现落点以本仓库为准。
2. **当前 `server` 只连了一个库 `operations`**（122.51.175.128:23306）。经确认：**`operations` 即需求所说的 `website` 后台库**，后台管理表建于此。
3. **`zhouyi`（APP 业务库）尚无连接信息**，用户将稍后提供。本期先搭好「双数据源框架」，zhouyi 依赖功能（驾驶舱/用户/订单分析）待连接信息到位再接真实表。
4. **XiaoMaYi-EleVue-v1.1.0 内不含 Vue 前端源码**（只有 Java 后端模块）。经确认：**由我按 XiaoMaYi 架构新建 Vue3 前端**。
5. 技术栈与 XiaoMaYi 同源（Spring Boot 3 + MyBatis-Plus + Sa-Token + Druid + Redis），扩展无障碍；新增模块全部走 `/api/admin/**`，与现有 `/api/**` C 端接口物理隔离。

### 1.3 现有 API 与编码规范（必须沿用）

- **统一响应**：`Result<T>`，成功 `code=0`（`Result.success(...)`），失败 `Result.error(code,msg)`；含 `timestamp`。
- **接口前缀**：`/api/**`（如 `/api/user/login`）。后台模块统一新增 `/api/admin/**`。
- **鉴权**：Sa-Token `StpUtil`，请求头 `Authorization`，Token 存 Redis（库 5）。
- **实体**：MyBatis-Plus，`@TableName` / `@TableId(IdType.AUTO)` / `@TableField(fill=INSERT|INSERT_UPDATE)`。
- **作者标注**：现有类 `@author FortuneTelling @since 1.0.0`，新增类沿用风格。

## 3. 前端分析（XiaoMaYi-EleVue-v1.1.0）

- 该框架是「Vue3 + TypeScript + Vite + ElementPlus + SpringBoot3 + MyBatisPlus」前后端分离方案，含完善 RBAC（按钮级权限）、多主题、代码生成器。
- **但本仓库内仅含其后端 Java 模块**（`xiaomayi-admin`、`xiaomayi-web`、`xiaomayi-system` 等），`static/` 下只有零散 css/js/image，**无 Vue 工程源码**（无 `package.json`、无 `.vue`、无 `vite.config`）。
- 结论：无现成前端组件可直接复用。**按 XiaoMaYi 的架构约定（Vue3+TS+Vite+ElementPlus+Pinia+ECharts、路由/权限/请求封装/布局体系）新建 `admin-web` 前端工程**，详见《XiaoMaYi-EleVue资源分析.md》与技术设计方案。

## 5. 风险与约束（与需求一致）

1. **不改 C 端**：不动 `/api/user`、`/api/game` 等现有接口与 `com.ydzz` 现有业务逻辑，仅新增 `com.ydzz.admin.*` 与 `/api/admin/**`。
2. **不臆造**：zhouyi 真实表结构未知前，相关查询以策划案字段为**预设**，拿到库后以真实表为准校正，绝不用模拟数据替代真实数据。
3. **双库隔离**：后台写操作只落 operations；zhouyi 一律只读。
4. **安全**：后台账号独立、BCrypt 加密、后端 `@SaCheckPermission` 校验（不只靠前端隐藏菜单）、MyBatis-Plus 参数化查询防注入、操作审计。
5. **待用户提供**：zhouyi 连接信息（host/port/user/pass/库名）及（可选）真实表结构 DDL。
