## 七、部署方式

本项目采用 **「开发分离、部署合一」**。

### 方式一（推荐）：单 jar 一体化 —— 前端打进后端，一个端口
1. 在 website(operations) 库执行 `server/src/main/resources/sql/admin_init.sql`。
2. 确保 MySQL（operations + zhouyi）、Redis 可用。
3. `cd server && mvn clean package -Pwith-frontend -DskipTests` —— 该 Profile 自动构建 admin-web 并打入 jar 的 `static/admin/`。
4. `java -jar target/website_server-1.0.0.jar`（端口 8660）。
   - 后台页面：`http://localhost:8660/admin/`
   - 接口：`http://localhost:8660/api/admin/**`
   - Swagger：`http://localhost:8660/swagger-ui.html`
5. 不带 `-Pwith-frontend` 的 `mvn package` 仍只打后端，原有 Docker/jar 部署不受影响。

实现要点：前端 UI 统一挂 `/admin/` 前缀，与 C 端 `/**`、接口 `/api/admin/**` 物理隔离；`WebConfig` 已放行 `/admin/**`（不经过 C 端登录拦截器）；`AdminWebMvcConfig` 通过自定义静态资源处理器（`PathResourceResolver` 回退）实现 SPA history 路由——真实文件直出、非文件路径回退 `index.html`，并将 `/admin` 重定向到 `/admin/`（兼容 Spring Boot 3 PathPattern 解析器，避免非法 `**` 通配映射）。

### 方式二：前后端独立部署（Nginx）
将 `admin-web/dist/` 交由 Nginx 托管，`/api` 反代到后端 8660；history 模式配置 `try_files $uri $uri/ /index.html`。

### 开发态（带 HMR）
- 根目录 `start-dev.bat` 一键拉起后端(8660)+前端(9660)；访问 `http://localhost:9660/`（`/api` 已代理）。

## 九、后续扩展建议

1. **补齐占位模块后端接口**：订单/商品/权益/支付异常监控、Banner/公告等，对应实体与 Mapper 多已就绪（`Orders`/`Payment`/`Refund`/`UserBenefit`/`Product` 等），按现有 Controller 范式补 Service+Controller 即可点亮占位页。
2. **驾驶舱字段以真实 zhouyi 表校正**：`DashboardMapper`/`UserAdminService` 中表名列名依据策划案预设，联调时按真实库微调 SQL。
3. **前端构建优化**：dashboard 因引入完整 echarts 体积偏大，可改为按需引入 `echarts/core` 降低产物体积。
4. **角色细粒度授权落地**：当前仅超级管理员预置全权限，运营/客服/财务/分析等角色权限可在「角色管理-分配权限」按策略配置。
5. **安全加固**：生产环境关闭 Swagger、限制 Druid 监控、Token 过期与并发策略按需收紧。
